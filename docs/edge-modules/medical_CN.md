# 医疗与健康模块——WiFi-DensePose边缘智能

> 使用WiFi信号进行非接触式健康监测。无需可穿戴设备,无需摄像头——只需一个ESP32传感器读取从人体反射的WiFi信号,以检测呼吸问题、心律问题、行走困难和癫痫。

## 重要免责声明

这些模块是**研究工具,而非FDA批准的医疗设备**。它们应该补充——而非替代——专业医疗监测。从WiFi CSI衍生的生命体征本质上比临床仪器(ECG、脉搏血氧仪、呼吸带)更嘈杂。误报和漏报将会发生。在根据警报采取行动之前,始终使用临床级设备验证发现。

## 模块

### 睡眠呼吸暂停检测(`med_sleep_apnea.rs`)

**功能**:从主机CSI管道监测呼吸频率,并检测当呼吸在连续10秒以上降至4 BPM以下时,表明呼吸暂停发作。它跟踪所有发作并计算呼吸暂停-低通气指数(AHI)——每小时监测睡眠时间的呼吸暂停事件数。AHI是睡眠呼吸暂停严重程度的临床标准指标。

**临床基础**:阻塞性和中枢性睡眠呼吸暂停定义为气流停止10秒或更长时间。该模块使用4 BPM的呼吸频率阈值(基本上是接近零的呼吸),并具有10秒发作延迟以确认停止是持续的。AHI严重程度分类:< 5正常,5-15轻度,15-30中度,> 30重度。

**工作原理**:
1. 每秒检查呼吸BPM是否低于4.0
2. 增加连续低呼吸计数器
3. 在连续10秒后,宣布呼吸暂停开始(回溯到呼吸首次下降时)
4. 当呼吸恢复到4 BPM以上时,记录发作及其持续时间
5. 每5分钟,计算AHI = (总发作数) / (监测小时数)
6. 仅在检测到存在时监测;如果受试者在呼吸暂停期间离开,则结束发作

#### API

| 项目 | 类型 | 描述 |
|------|------|-------------|
| `SleepApneaDetector` | struct | 主检测器状态 |
| `SleepApneaDetector::new()` | `const fn` | 创建零状态检测器 |
| `process_frame(breathing_bpm, presence, variance)` | method | 以~1 Hz处理一帧;返回事件切片 |
| `ahi()` | method | 当前AHI值 |
| `episode_count()` | method | 记录的呼吸暂停发作总数 |
| `monitoring_seconds()` | method | 存在激活的总秒数 |
| `in_apnea()` | method | 当前是否在呼吸暂停发作中 |
| `APNEA_BPM_THRESH` | const | 4.0 BPM -- 低于此值计为呼吸暂停 |
| `APNEA_ONSET_SECS` | const | 10秒 -- 宣布呼吸暂停的最短持续时间 |
| `AHI_REPORT_INTERVAL` | const | 300秒(5分钟) -- AHI重新计算的频率 |
| `MAX_EPISODES` | const | 256 -- 每次会话存储的最大发作数 |

#### 发出的事件

| 事件ID | 常量 | 值 | 临床意义 |
|----------|----------|-------|-----------------|
| 100 | `EVENT_APNEA_START` | 当前呼吸BPM | 呼吸已停止或降至4 BPM以下超过10秒 |
| 101 | `EVENT_APNEA_END` | 持续时间(秒) | 呼吸暂停发作后呼吸已恢复 |
| 102 | `EVENT_AHI_UPDATE` | AHI分数(事件/小时) | 周期性严重程度指标;>5 = 轻度,>15 = 中度,>30 = 重度 |

#### 状态机

```
                          存在丢失
    [监测中] -----> [未监测] (无事件,计数器暂停)
         |                    |
         | bpm < 4.0          | 恢复存在
         v                    v
    [低呼吸计数器]  [监测中]
         |
         | count >= 10s
         v
    [呼吸暂停中] ---------> [发作结束] (bpm >= 4.0 或 存在丢失)
         |                      |
         |                      v
         |               [记录发作,发出APNEA_END]
         |
         +-- 发出APNEA_START (一次)
```

#### 示例用法

```rust
use wifi_densepose_wasm_edge::med_sleep_apnea::*;

let mut detector = SleepApneaDetector::new();

// 正常呼吸 -- 无事件
let events = detector.process_frame(14.0, 1, 0.1);
assert!(events.is_empty());

// 模拟呼吸暂停:连续15秒输入低BPM
for _ in 0..15 {
    let events = detector.process_frame(1.0, 1, 0.1);
    for &(event_id, value) in events {
        match event_id {
            EVENT_APNEA_START => println!("检测到呼吸暂停! BPM: {}", value),
            _ => {}
        }
    }
}
assert!(detector.in_apnea());

// 恢复正常呼吸
let events = detector.process_frame(14.0, 1, 0.1);
for &(event_id, value) in events {
    match event_id {
        EVENT_APNEA_END => println!("呼吸暂停在{}秒后结束", value),
        _ => {}
    }
}

println!("发作数: {}", detector.episode_count());
println!("AHI: {:.1}", detector.ahi());
```

#### 教程:设置卧室睡眠监测

1. **ESP32放置**:将ESP32-S3安装在距离床1-2米的墙壁或天花板上,位于胸部高度。传感器应与睡眠区域有视线。避免放置在金属物体或产生CSI干扰的移动风扇附近。

2. **WiFi路由器**:确保稳定的WiFi AP在范围内。ESP32监测从人体反射的WiFi信号的CSI(信道状态信息)。AP应位于床的传感器对面,以获得最佳身体反射捕获。

3. **固件配置**:使用启用的第2层边缘处理(提供呼吸BPM)刷写ESP32固件。睡眠呼吸暂停WASM模块作为第3层算法在第2层生命体征输出之上运行。

4. **阈值调整**:默认4 BPM阈值是保守的(接近完全停止)。对于更敏感的检测器,降低到6-8 BPM,但预期浅呼吸会有更多误报。10秒发作延迟符合临床呼吸暂停定义。

5. **读取AHI结果**:AHI每5分钟发出一次。整夜(7-8小时)后,最终AHI值代表夜间严重程度。与临床阈值比较:< 5(正常),5-15(轻度),15-30(中度),> 30(重度)。

6. **限制**:基于WiFi的呼吸检测在受试者相对静止(睡眠)时效果最佳。辗转反侧可能导致呼吸检测的瞬时丢失,这可能掩盖或错误触发呼吸暂停事件。单夜研究应始终通过临床多导睡眠图确认。

### 心律失常检测(`med_cardiac_arrhythmia.rs`)

**功能**：监测来自主机CSI管道的心率，并检测四种类型的心律异常：心动过速（持续快心率）、心动过缓（持续慢心率）、漏跳（突然心率下降）和HRV异常（心率变异性超出正常范围）。

**临床基础**：心动过速定义为HR > 100 BPM持续10+秒。心动过缓是HR < 50 BPM持续10+秒（使用50 BPM阈值而不是典型的60 BPM，以考虑CSI测量噪声并避免运动员自然低静息HR的误报）。漏跳被检测为与运行平均值相比>30%的下降。HRV通过RMSSD（连续差异的均方根）评估，使用加宽的正常带（10-120 ms等效）来考虑与ECG相比更粗糙的CSI衍生HR测量。

**工作原理**：
1. 维持心率的指数移动平均（EMA），alpha=0.1
2. 跟踪连续秒数超过100 BPM（心动过速）或低于50 BPM（心动过缓）
3. 在异常范围内连续10秒后，发出相应警报
4. 计算与EMA的分数下降以检测漏跳
5. 维持30秒环形缓冲区用于RMSSD计算的连续HR差异
6. RMSSD从BPM单位转换为近似ms等效值（比例因子~17）
7. 所有警报都有30秒冷却期以防止事件泛滥
8. 无效读数（< 1 BPM或NaN）被静默忽略以防止污染

#### API

| 项目 | 类型 | 描述 |
|------|------|-------------|
| `CardiacArrhythmiaDetector` | struct | 主检测器状态 |
| `CardiacArrhythmiaDetector::new()` | `const fn` | 创建零状态检测器 |
| `process_frame(hr_bpm, phase)` | method | 以~1 Hz处理一帧；返回事件切片 |
| `hr_ema()` | method | 当前EMA心率 |
| `frame_count()` | method | 处理的总帧数 |
| `TACHY_THRESH` | const | 100.0 BPM |
| `BRADY_THRESH` | const | 50.0 BPM |
| `SUSTAINED_SECS` | const | 10秒 |
| `MISSED_BEAT_DROP` | const | 0.30（与EMA相比30%下降） |
| `HRV_WINDOW` | const | 30秒 |
| `RMSSD_LOW` / `RMSSD_HIGH` | const | 10.0 / 120.0 ms（为CSI加宽） |
| `COOLDOWN_SECS` | const | 30秒 |

#### 发出的事件

| 事件ID | 常量 | 值 | 临床意义 |
|----------|----------|-------|-----------------|
| 110 | `EVENT_TACHYCARDIA` | 当前HR（BPM） | 心率持续超过100 BPM达10+秒 |
| 111 | `EVENT_BRADYCARDIA` | 当前HR（BPM） | 心率持续低于50 BPM达10+秒 |
| 112 | `EVENT_MISSED_BEAT` | 当前HR（BPM） | 从运行平均值突然下降>30% |
| 113 | `EVENT_HRV_ANOMALY` | RMSSD值（ms） | 心率变异性超出10-120 ms正常范围 |

#### 状态机

心脏模块没有正式的状态机——它使用带有冷却定时器的独立检测器：

```
每一帧：
  1. 滴答冷却（4个独立定时器）
  2. 拒绝无效输入（< 1 BPM或NaN）
  3. 更新EMA（alpha = 0.1）
  4. 更新RR差异环形缓冲区
  5. 检查心动过速（HR > 100持续10+连续秒）
  6. 检查心动过缓（HR < 50持续10+连续秒）
  7. 检查漏跳（与EMA相比>30%下降）
  8. 检查HRV异常（RMSSD超出10-120 ms，需要完整30s窗口）
  9. 每次检查都遵守自己的30秒冷却期
```

#### 配置

| 参数 | 默认值 | 临床范围 | 描述 |
|-----------|---------|----------------|-------------|
| `TACHY_THRESH` | 100.0 | 90-120 BPM | 心动过速的HR阈值 |
| `BRADY_THRESH` | 50.0 | 40-60 BPM | 心动过缓的HR阈值 |
| `SUSTAINED_SECS` | 10 | 5-30 s | 警报所需的连续秒数 |
| `MISSED_BEAT_DROP` | 0.30 | 0.20-0.40 | 标记漏跳的分数HR下降 |
| `RMSSD_LOW` | 10.0 | 5-20 ms | 最小正常RMSSD |
| `RMSSD_HIGH` | 120.0 | 80-150 ms | 最大正常RMSSD |
| `EMA_ALPHA` | 0.1 | 0.05-0.2 | EMA平滑系数 |
| `COOLDOWN_SECS` | 30 | 10-60 s | 重复警报之间的最短时间 |

#### 示例用法

```rust
use wifi_densepose_wasm_edge::med_cardiac_arrhythmia::*;

let mut detector = CardiacArrhythmiaDetector::new();

// 正常心率 — 无事件
for _ in 0..60 {
    let events = detector.process_frame(72.0, 0.0);
    assert!(events.is_empty() || events.iter().all(|&(t, _)| t == EVENT_HRV_ANOMALY));
}

// 持续心动过速
for _ in 0..15 {
    let events = detector.process_frame(120.0, 0.0);
    for &(event_id, value) in events {
        if event_id == EVENT_TACHYCARDIA {
            println!("心动过速警报！HR: {} BPM", value);
        }
    }
}
```

### 呼吸窘迫检测(`med_respiratory_distress.rs`)

**功能**：从主机CSI管道检测四种类型的呼吸异常：呼吸急促（快速呼吸）、呼吸困难（高幅度方差）、潮式呼吸（ crescendo-decrescendo呼吸模式），以及0-100的综合呼吸窘迫严重程度评分。

**临床基础**：呼吸急促在临床上定义为成人> 20 BPM。该模块使用25 BPM的阈值（更保守）以减少来自固有噪声更大的CSI衍生呼吸率的误报。呼吸困难被检测为相对于学习基线的幅度方差增加3倍。潮式呼吸是一种病理呼吸模式，周期为30-90秒，通常与心力衰竭和神经系统疾病相关。该模块通过呼吸幅度包络的自相关来检测它。

**工作原理**：
1. 维持120秒呼吸BPM环形缓冲区用于自相关分析
2. 维持60秒幅度方差环形缓冲区
3. 在最初60秒内学习基线方差（Welford在线均值）
4. 检查呼吸急促：呼吸率> 25 BPM持续8+秒
5. 检查呼吸困难：当前方差> 3x基线方差
6. 检查潮式呼吸：30-90秒滞后范围内的显著自相关峰值
7. 每30秒基于以下因素计算综合窘迫评分（0-100）：与正常（16 BPM中心）的速率偏差、方差比率、呼吸急促标志和最近的潮式呼吸检测
8. NaN输入从环形缓冲区中排除以防止污染

#### API

| 项目 | 类型 | 描述 |
|------|------|-------------|
| `RespiratoryDistressDetector` | struct | 主检测器状态 |
| `RespiratoryDistressDetector::new()` | `const fn` | 创建零状态检测器 |
| `process_frame(breathing_bpm, phase, variance)` | method | 以~1 Hz处理一帧；返回事件切片 |
| `last_distress_score()` | method | 最近的综合评分（0-100） |
| `frame_count()` | method | 处理的总帧数 |
| `TACHYPNEA_THRESH` | const | 25.0 BPM（保守；临床为20 BPM） |
| `SUSTAINED_SECS` | const | 8秒 |
| `LABORED_VAR_RATIO` | const | 3.0x基线 |
| `CS_LAG_MIN` / `CS_LAG_MAX` | const | 30 / 90秒（潮式呼吸周期范围） |
| `CS_PEAK_THRESH` | const | 0.35（归一化自相关） |
| `BASELINE_SECS` | const | 60秒（学习期） |
| `COOLDOWN_SECS` | const | 20秒 |

#### 发出的事件

| 事件ID | 常量 | 值 | 临床意义 |
|----------|----------|-------|-----------------|
| 120 | `EVENT_TACHYPNEA` | 当前呼吸BPM | 呼吸率持续超过25 BPM达8+秒 |
| 121 | `EVENT_LABORED_BREATHING` | 方差比率 | 呼吸努力> 3x基线；可能呼吸窘迫 |
| 122 | `EVENT_CHEYNE_STOKES` | 周期（秒） | Crescendo-decrescendo呼吸模式；与心力衰竭相关 |
| 123 | `EVENT_RESP_DISTRESS_LEVEL` | 评分0-100 | 综合严重程度：0-20正常，20-50轻度，50-80中度，80-100严重 |

#### 状态机

呼吸窘迫模块使用带有冷却的独立检测器轨道，而不是单个状态机：

```
每一帧：
  1. 滴答冷却（3个独立定时器）
  2. 跳过环形缓冲区更新的NaN输入
  3. 更新呼吸BPM环形缓冲区（120s）和方差环形缓冲区（60s）
  4. 在最初60秒内学习基线方差（Welford）
  5. 呼吸急促检查：BPM > 25持续8+连续秒
  6. 呼吸困难：当前方差均值 > 3x基线（基线期后）
  7. 潮式呼吸：30-90s滞后范围内自相关峰值 > 0.35（需要完整120s缓冲区）
  8. 每30秒发出综合窘迫评分
```

| `CS_LAG_MIN` / `CS_LAG_MAX` | 30 / 90 | 20-120 s | 潮式呼吸周期搜索范围 |

| `DISTRESS_REPORT_INTERVAL` | 30 | 10-60 s | 综合评分发出的频率 |
| `COOLDOWN_SECS` | 20 | 10-60 s | 重复警报之间的最短时间 |

#### 教程：设置ICU/病房监测

1. **放置**：将ESP32安装在床脚或直接在患者上方的天花板上。传感器需要从患者躯干清晰的WiFi信号反射。

2. **基线学习**：模块在首次激活时自动学习60秒基线方差。确保患者在此校准期间正常呼吸。如果患者在模块启动时已经处于窘迫状态，基线将被扭曲，呼吸困难检测将不可靠。

3. **潮式呼吸检测**：需要至少120秒的数据才能开始自相关分析。30-90秒周期搜索范围涵盖临床记录的潮式呼吸周期范围。实际上，检测通常在3-4分钟的监测后变得可靠。

4. **窘迫评分解释**：综合评分（0-100）结合了四个因素：与正常的速率偏差、方差比率、呼吸急促存在和潮式呼吸检测。分数超过50需要临床关注。超过80表明急性窘迫。

### 步态分析(`med_gait_analysis.rs`)

**功能**：从CSI相位方差周期性中提取步态参数，以评估 mobility 和跌倒风险。检测步频、步态不对称（跛行）、步幅变异性、拖曳步态模式（与帕金森病相关）、慌张步态（不自主加速），并计算0-100的综合跌倒风险评分。

**临床基础**：健康成年人的正常行走步频为80-120步/分钟。拖曳步态（>140步/分钟，低能量）是帕金森病和其他神经系统疾病的特征。慌张步态（不自主步频加速）是帕金森氏特征。步态不对称（左右步间隔比率偏离1.0超过15%）表明跛行或肌肉骨骼问题。高步幅变异性（变异系数）是老年患者跌倒风险的强预测因子。

**工作原理**：
1. 维持60秒相位方差和运动能量环形缓冲区
2. 将步骤检测为相位方差信号中的局部最大值（峰谷比> 1.5）
3. 在64条目缓冲区中记录步间隔
4. 每10秒计算：步频（60 / 平均步间隔）、不对称（奇数/偶数步间隔比率）、变异性（变异系数）
5. 跟踪6个报告周期的步频历史以检测慌张步态
6. 当步频> 140且运动能量低时标记拖曳
7. 慌张步态检测为步频加速> 1.5步/分钟/秒
8. 跌倒风险评分（0-100）是以下因素的加权综合：异常步频（25%）、不对称（25%）、变异性（25%）、低能量（15%）、慌张步态（10%）

#### 状态机

步态分析器在周期性报告周期上运行：

```
连续（每帧）：
  - 将方差和能量推入环形缓冲区
  - 检测步骤峰值（方差局部最大值 > 1.5x邻居）
  - 记录步间隔

每REPORT_INTERVAL（10s），如果检测到>= 4步：
  1. 计算步频、不对称、变异性
  2. 发出EVENT_STEP_CADENCE
  3. 如果不对称>阈值：发出EVENT_GAIT_ASYMMETRY
  4. 如果步频> 140且能量< 0.3：发出EVENT_SHUFFLING_DETECTED
  5. 如果步频在3个周期内加速> 1.5/s：发出EVENT_FESTINATION
  6. 计算并发出EVENT_FALL_RISK_SCORE
  7. 重置步缓冲区为下一个窗口
```

#### 教程：设置走廊步态监测

1. **放置**：将ESP32安装在走廊或通道的墙上，腰部高度。步行路径应在传感器视野内3-5米长。将WiFi AP放置在走廊的另一端以获得最佳身体反射。

2. **校准**：步骤检测器依赖于相位方差的周期性峰值。1.5的`STEP_PEAK_RATIO`适用于大多数地板表面。在地毯（会减弱冲击信号）上，考虑降低到1.2。在硬地板穿鞋时，1.5-2.0是合适的。

3. **临床背景**：跌倒风险评分对于纵向监测最有用。单次读数提供快照，但跟踪几天/几周的趋势揭示渐进性移动能力下降。上升的跌倒风险评分（例如，一个月内从20上升到40）即使个别读数低于"高风险"阈值也需要临床评估。

4. **限制**：在1 Hz计时器速率下，模块无法通过直接峰值计数检测高于~60步/分钟的步频。对于更高的步频，步骤检测依赖于主机的更高速率CSI处理来预计算方差峰值。>140步/分钟的拖曳检测需要主机以高于1 Hz的速率提供步骤级方差数据。

### 癫痫检测(`med_seizure_detect.rs`)

**功能**：通过识别3-8 Hz频带中的持续高能量节律性运动来检测强直-阵挛（大发作）癫痫。区分癫痫与跌倒（单次冲动后静止）和震颤（较低幅度，更高规律性）。跟踪癫痫阶段：强直（持续肌肉僵硬）、阵挛（节律性抽搐）和发作后（运动突然停止）。

**临床基础**：强直-阵挛癫痫有特征性进展：(1) 强直阶段，持续肌肉僵硬导致高运动能量，低方差，持续10-20秒；(2) 阵挛阶段，3-8 Hz的节律性抽搐，持续30-60秒；(3) 发作后阶段，运动突然停止和深度无反应。跌倒产生短暂（<10帧）高能量尖峰，然后静止。震颤的幅度低于癫痫级别的抽搐。

**工作原理**：
1. 以~20 Hz帧率运行（高于其他模块）以进行节律检测
2. 维持100帧运动能量和幅度环形缓冲区
3. 状态机进展：监测 -> 可能发作 -> 强直/阵挛 -> 发作后 -> 冷却
4. 发作需要10+连续帧的高运动能量（>2.0归一化）
5. 跌倒歧视：如果高能量持续< 10帧然后下降，则分类为跌倒并忽略
6. 强直阶段：高能量，低方差（< 0.5）
7. 阵挛阶段：通过幅度缓冲区的自相关检测2-7帧周期（20 Hz采样时3-8 Hz）
8. 发作后：运动降至0.2以下持续40+连续帧
9. 发作后，200帧冷却防止重新触发
10. 存在必须活跃；存在丢失重置状态机

#### API

| 项目 | 类型 | 描述 |
|------|------|-------------|
| `SeizureDetector` | struct | 主检测器状态 |
| `SeizureDetector::new()` | `const fn` | 创建零状态检测器 |
| `process_frame(phase, amplitude, motion_energy, presence)` | method | 以~20 Hz处理；返回事件切片 |
| `phase()` | method | 当前`SeizurePhase`枚举值 |
| `seizure_count()` | method | 检测到的癫痫发作总数 |
| `frame_count()` | method | 处理的总帧数 |
| `SeizurePhase` | enum | Monitoring, PossibleOnset, Tonic, Clonic, PostIctal, Cooldown |
| `HIGH_ENERGY_THRESH` | const | 2.0（归一化） |
| `TONIC_MIN_FRAMES` | const | 20帧（20 Hz时1秒） |
| `CLONIC_PERIOD_MIN` / `MAX` | const | 2 / 7帧（20 Hz时3-8 Hz） |
| `POST_ICTAL_MIN_FRAMES` | const | 40帧（20 Hz时2秒） |
| `COOLDOWN_FRAMES` | const | 200帧（20 Hz时10秒） |

#### 发出的事件

| 事件ID | 常量 | 值 | 临床意义 |
|----------|----------|-------|-----------------|
| 140 | `EVENT_SEIZURE_ONSET` | 运动能量 | 检测到癫痫活动；需要立即临床关注 |
| 141 | `EVENT_SEIZURE_TONIC` | 持续时间（帧） | 识别强直阶段；持续僵硬 |
| 142 | `EVENT_SEIZURE_CLONIC` | 周期（帧） | 识别阵挛阶段；具有检测到的周期性的节律性抽搐 |
| 143 | `EVENT_POST_ICTAL` | 1.0 | 发作后阶段；癫痫后运动已停止 |

#### 状态机

```
                    存在丢失（从任何活跃状态）
                    +-----------------------------------------+
                    v                                         |
[Monitoring] --> [PossibleOnset] --> [Tonic] --> [Clonic] --> [PostIctal] --> [Cooldown]
      ^              |    |              |                         |              |
      |              |    |              +------> [PostIctal] -----+              |
      |              |    |                (能量下降时直接)               |
      |              |    +--------> [Clonic]                                    |
      |              |            (跳过强直)                                   |
      |              |                                                           |
      |              +-- 超时（200帧） --> [Monitoring]                   |
      |              +-- 跌倒（<10帧） -----> [Monitoring]                   |
      |                                                                          |
      +------ 冷却到期（200帧） ------------------------------------+
```

转换：
- **监测 -> 可能发作**：10+帧运动能量 > 2.0
- **可能发作 -> 强直**：低能量方差 + 高能量（肌肉僵硬模式）
- **可能发作 -> 阵挛**：节律性自相关峰值 + 幅度高于震颤底线
- **可能发作 -> 监测**：10帧内能量下降（跌倒）或200帧超时
- **强直 -> 阵挛**：能量方差增加并检测到节律
- **强直 -> 发作后**：运动能量降至0.2以下持续40+帧
- **阵挛 -> 发作后**：运动能量降至0.2以下持续40+帧
- **发作后 -> 冷却**：发作后40帧后
- **冷却 -> 监测**：200帧后（10秒）

#### 示例用法

```rust
use wifi_densepose_wasm_edge::med_seizure_detect::*;

let mut detector = SeizureDetector::new();

// 正常运动 — 无癫痫
for _ in 0..200 {
    let events = detector.process_frame(0.0, 0.5, 0.3, 1);
    assert!(events.is_empty());
}
assert_eq!(detector.phase(), SeizurePhase::Monitoring);

// 强直阶段：持续高能量，低方差
for _ in 0..50 {
    let events = detector.process_frame(0.0, 2.0, 3.0, 1);
    for &(event_id, value) in events {
        match event_id {
            EVENT_SEIZURE_ONSET => println!("癫痫发作！能量：{}", value),
            EVENT_SEIZURE_TONIC => println!("强直阶段：{}帧", value),
            _ => {}
        }
    }
}

// 发作后：突然停止
for _ in 0..100 {
    let events = detector.process_frame(0.0, 0.05, 0.05, 1);
    for &(event_id, _) in events {
        if event_id == EVENT_POST_ICTAL {
            println!("检测到发作后阶段 — 患者需要立即评估");
        }
    }
}
```

#### 教程：设置癫痫监测

1. **放置**：将ESP32安装在天花板上，直接位于床或监测区域上方。癫痫检测需要对身体运动的最高灵敏度，因此最小化与患者的距离。确保传感器视野中没有其他人或移动物体（宠物、窗帘、风扇）。

2. **帧率**：与以1 Hz运行的其他医疗模块不同，癫痫检测器期望~20 Hz帧输入以准确检测3-8 Hz频带中的节律。加载此模块时，确保主机固件配置为高速率CSI处理。

3. **灵敏度调整**：2.0的`HIGH_ENERGY_THRESH`和10的`ONSET_MIN_FRAMES`平衡了灵敏度与误报。在安静的卧室环境中，这些默认值效果良好。在更嘈杂的环境（共享病房、附近设备振动）中，考虑将`HIGH_ENERGY_THRESH`提高到2.5-3.0。

4. **跌倒与癫痫歧视**：模块自动区分跌倒（短暂能量尖峰< 10帧）和癫痫（持续能量）。如果患者已知有跌倒风险，考虑并行运行步态分析模块以进行补充监测。

5. **响应协议**：当`EVENT_SEIZURE_ONSET`触发时，立即通知临床人员。`EVENT_POST_ICTAL`事件表示活动性癫痫已结束，患者进入发作后状态 — 他们需要评估但不再处于抽搐阶段。

### 阈值原理

几个阈值与严格的临床标准不同。这是有意的：

- **WiFi CSI不是ECG/脉搏血氧仪**。信噪比更低，因此阈值被加宽以减少误报，同时保持临床相关性。
- **保守阈值优先考虑特异性而非敏感性**。在非临床级系统中，错过警报比警报疲劳更可取。
- **所有阈值都是编译时常量**。要针对特定部署进行调整，修改每个模块文件顶部的常量并重新编译。

## 安全考虑

1. **不能替代医疗设备**。这些模块是研究/辅助工具。它们未经临床试验验证，未获得FDA/CE认证。永远不要将它们作为患者监测的唯一来源。

2. **误报率**。WiFi CSI受环境因素影响：移动物体（风扇、宠物、窗帘）、多径变化（开门、附近有人行走）和电磁干扰。典型家庭环境中预期误报率为5-15%，受控临床环境中为1-5%。

3. **漏报率**。保守阈值意味着一些边界条件可能不会触发警报。具体来说：
   - 呼吸过缓（12-20 BPM降至12-4 BPM）未直接标记 — 仅检测低于4 BPM的呼吸暂停
   - 轻度心动过速（100-120 BPM）被检测，但10秒持续要求意味着短暂发作被错过
   - 无强运动成分的低幅度癫痫可能未超过能量阈值

4. **影响准确性的环境因素**：
   - **多人环境**：所有模块假设单一受试者。传感器视野中的多人会损坏读数。
   - **距离**：CSI灵敏度随距离下降。将传感器放置在离受试者2米内。
   - **障碍物**：传感器和受试者之间的厚墙、金属家具和大型水体（水族馆）会降低性能。
   - **WiFi拥塞**：同一信道上的繁重WiFi流量增加CSI测量中的噪声。

5. **电源和连接**：ESP32必须维持连续WiFi连接以进行CSI监测。电源丢失或WiFi断开将静默停止所有监测。考虑为关键应用使用UPS电源和冗余AP放置。

6. **数据隐私**：这些模块处理健康相关数据。在临床或家庭护理环境中部署时，确保符合HIPAA、GDPR或当地健康数据法规。CSI数据和发出的事件应在传输和静止时加密。
